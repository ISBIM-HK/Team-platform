§ 一、用户故事

7 个核心用户故事，覆盖团队成员和 PM 两类角色。

团队成员
作为团队成员，我希望平台能自动从 GitLab、飞书等工具拉取我的日常工作痕迹，这样我就不用手动记录每天做了什么。
消除手动上传工作的行政开销，让成员专注在真正有价值的事情上。
团队成员
作为团队成员，我希望 AI 能根据我的工作痕迹自动生成日报和周报，我只需审阅确认即可。
将写周报的时间从 30 分钟压缩到 2 分钟审阅，提升团队效率。
团队成员
作为团队成员，我希望有一个个人 AI 助手，能帮我查任务、记手动工作、搜文档，像一个随时在线的工作伙伴。
降低上下文切换成本，所有工作信息一个入口获取。
团队成员
作为团队成员，我希望在 Kanban 上看到全团队的任务状态，了解同事在做什么，方便我 pull 模式认领任务。
透明化团队工作状态，促进自主任务认领和协作。
PM
作为 PM，我希望看到团队聚合的工作负载视图，了解每个人的任务分布和进度，无需逐一询问。
从"逐一问进度"变为"看面板决策"，节省 PM 和团队双方的时间。
PM
作为 PM，我希望 AI 能智能推荐任务分发给合适的成员，并给出理由，我确认后执行。
提升任务分发效率，让 PM 从"派活"转向"审核 AI 建议"。
PM
作为 PM，我希望一键生成对外汇报摘要，将多个周报和关键任务自动整合为面向客户的简报。
将对外汇报准备时间从 1 小时压缩到 5 分钟，提升客户沟通效率。

▶ Task（任务） 聚合根
字段类型说明
idUUID主键
tenant_idUUID所属租户
project_idUUID (required)所属项目（每个任务必属一个项目；兜底"未分类"项目）
titleString (required)任务标题
descriptionText (markdown)任务描述
statusEnumtodo / in_progress / blocked / review / done / archived
priorityInt0=low, 1=normal, 2=high, 3=urgent
owner_user_idUUID (nullable)负责人，可空（pull 模式未认领）
created_byString'user:<uuid>' 或 'ai_auto:<reason>'
source_event_idUUID (nullable)关联的 events_cache 记录
parent_task_idUUID (nullable)父任务，支持子任务
tagsJSONB标签数组
due_dateDate (nullable)截止日期
estimated_hoursNumeric (nullable)预估工时
created_atDateTime创建时间
updated_atDateTime更新时间
  • 状态流转必须遵循状态机：todo → in_progress → review → done；review 可退回 in_progress；blocked 仅从 in_progress 进入，仅能退回 in_progress（不直接跳 done，避免"绕过 review"）
  • owner_user_id 为空时任务处于"未认领"状态；pull 模式认领走乐观锁（UPDATE ... WHERE owner_user_id IS NULL，affected rows=0 时返 409 Conflict）
  • created_by 为 'ai_auto' 时需有对应的 ai_suggestions 记录
  • project_id 必填——decompose 落地时建/选项目；手动/历史任务进租户"未分类"项目
  • Task 聚合内含两个子实体（经聚合根访问）：TaskHistory（状态/字段变更流水，审计）与 TaskLink（任务关联：blocks / relates_to / duplicate_of）

§ 六、业务边界与规则

状态机、操作限制、权限矩阵、计算规则、业务规则汇总。

6.1 状态机

# 文字版(JS 渲染不出时的兜底,与 assets/state-machines.js 一致)
Task:      todo → in_progress;  in_progress → {review, done, blocked};
           review → {done, in_progress};  blocked → in_progress(仅此);  done → archived
AISuggestion: pending → {accepted, rejected, expired}（终态）
Integration:  active ⇄ error;  active/error → disabled;  any → expired(凭据过期)

6.5 业务规则汇总

  1. AI 不直接修改 Kanban 所有 AI 产出都生成 ai_suggestions，等用户 accept/reject。半年观察期后再讨论自动通道。
  2. 拆解与分配都只建议、不强制 任务拆解（目标→子任务）与分配建议（建议负责人）都走 ai_suggestions，人确认才落地——保留扁平 pull 模式（谁有空谁接），AI 不自动指派、不替人决策。这是与被否的"中心化派活"方案的根本区别。
  3. 低置信度过滤 confidence < 0.6 的建议不展示给用户，减少噪音。
  4. 事件缓存 append-only events_cache 只追加不修改，TTL 90 天自动清理。
  5. 凭据加密存储 第三方 token 用 Fernet 对称加密，主密钥 CRYPTO_KEY 从环境变量读，不入库不入仓库。
  6. 聊天内容绝对隐私 chat 内容仅本人可见，PM 也不能看别人 chat 原文。
  7. 资源不存在返回 404 不返回 403，避免泄露资源存在性信息。
  8. 写操作需 CSRF Token 所有非 GET 请求需要 X-CSRF-Token Header。
  9. 超预算降级 日预算超→降级模型（Pro→Flash）；月硬顶→暂停非必要 AI，保留个人助手。
  10. 迁移禁止破坏性操作 必须"加新列 → 双写过渡 → 删老列"分多次迁移。
  11. 阶段门槛不可逾越 每阶段 DoD 不达成绝不上线；用 2 周没人用的功能立刻砍。

    supports_webhook: bool = False

    async def authenticate(self, credential: dict) -> bool: ...

A.2 按需拉取编排

async def fetch_user_activity(user_id, window, sources=None):
    cache_key = (user_id, window, tuple(sources or "all"))
    if cached := cache.get(cache_key, max_age=300):  # 5 分钟缓存
        return cached

    adapters = [load_adapter(s) for s in sources or ALL_SOURCES]
    results = await asyncio.gather(
        *[a.fetch(user_id, since=now - window) for a in adapters],
        return_exceptions=True,  # 单源失败不影响其他
    )
    events = flatten_and_dedupe(results)
    upsert_events_cache(events)
    cache.set(cache_key, events)
    return events

触发器：用户操作 / 定时报告 / AI 助手对话开始 → Capture Orchestrator → asyncio.gather 并行拉各源 → events_cache → AI 处理。 不再有持续轮询：API 调用量下降 95%，LLM 成本只在实际需要产出时发生。

A.4 失败处理策略

  • 单次 fetch 异常 log warning，跳过该 source，其他 source 不受影响（asyncio.gather(return_exceptions=True)）。
  • 连续 3 次失败 integrations.enabled = False + 通知用户重新授权。
  • 全部 source 都挂 报告生成时明确说明"今天没拉到任何数据"，不脑补。

A.5 隐私与权限开关

每用户的 capture_preferences JSONB 控制：总开关（请假/敏感期暂停所有捕获）、单源开关（暂停某集成）、关键词屏蔽（含某些词的事件不入库）、数据可见性（仅本人 vs PM 可聚合）。每次写 events_cache 时检查。

rsync -a ./data/backups/ user@nas:/backups/teamplat/

⚠️ 容灾原则：DB 日备 + 附件周 rsync；加密主密钥额外备份（密码管理器 + 纸质双备份）；每季度真演练一次恢复——不演练的备份都是骗自己。

C.4 升级与回滚

# 回滚

C.5 关键技术决策

  • Postgres 不用 SQLite docker 一行命令起 Postgres，扩张时零迁移。
  • Caddy 不用 Nginx TLS 自动，配置 3 行。
  • 前端打包后由 FastAPI 静态托管 少一容器，同源无 CORS。
  • 不上 Prometheus/Grafana 6 人项目过度——结构化 JSON 日志 + /health + /metrics 即可。
  • 日志 structlog 结构化 JSON → ./logs/app-YYYY-MM-DD.jsonl；关键错误推送到飞书/钉钉 webhook。

附录 D：A → B 演进路径

决策日志里提了"B-ready 纪律"——这里展开具体是什么，以及什么时候切、怎么切。

D.1 切换触发信号

不基于用户数，基于以下真实压力信号（任一触发）：

信号阈值
单进程 CPU 持续高负载CPU > 70% 持续 1 周
AI 批处理拖累 web 响应P95 web 响应 > 1s

需异地灾备出现明确需求
events_cache 数据量大超 100GB

D.2 迁移顺序

A → 1. PG 升云托管/主从
  → 2. API 副本化 + LB + session 切 Redis
  → 3. 抽 capture workers 出独立容器
  → 4. events_cache 改 NATS JetStream + PG 归档（双写 1~2 周 → 切单写）
  → 5. 抽 AI processor 出独立容器
  → B 完成

每步独立上线 + 可独立回滚，绝不一次性大重构。

D.3 拆什么、不拆什么

模块A 阶段B 阶段改造成本
API + 前端单进程 FastAPI多副本 + LB几乎零
events_cachePG 一张表NATS JetStream + PG 归档中
Capture workers同进程函数独立 worker 容器 + 工作队列中
AI processor同步函数独立 worker + 任务队列中
DB单 PG主从/云托管几乎零
WebSocket内存路由Redis Pub/Sub 跨副本小

不拆的部分：数据模型、API 契约、前端、Auth、AI 业务逻辑——B 对前端完全透明。

D.4 B-ready 纪律（A 阶段 day 1 执行）

  1. 所有 capture adapter 实现统一接口 Abstract base class CaptureAdapter，每个源一个实现。
  2. AI 函数是纯函数 输入事件 → 输出结构化结果，无副作用。
  3. events_cache schema 严格 + 按时间分区 schema 不随实现变。
  4. DB 访问全经 repository 层 不直接在业务代码里写 SQL。
  5. Session 不依赖进程内存 先内存，接口预留 Redis。
  6. 配置驱动 config.yaml 管理阈值/模型/集成，可不重启切换。

附录 F：调度与通知

承载"提醒 / 汇报 / 开会通知 / PM 看进展"等自动行为——它们是定时任务，不是多 Agent 协作。

F.1 调度器选型

  • APScheduler（进程内） FastAPI 同进程起一个 AsyncIOScheduler，启动时读 scheduled_jobs 表注册触发，不另起常驻进程。
  • cron ≠ 持续轮询 cron 只触发内部计算/报告生成(定时产出);它不是对外部系统(GitLab/飞书)的持续轮询——外部数据仍按需拉取(§2.1)。"不持续轮询"针对的是外部 API,定时产出不违反该原则。
  • 不上 Celery / Arq 6 人 / 1 PM 规模过度——独立 broker + worker 容器是 A→B 的事（见附录 D）。
  • 状态即 DB 行 "谁有哪些待发提醒 / job 跑到哪了 / 去重" 全是 scheduled_jobs + notifications 表的行，不用文件、不用 agent 内存。

F.2 触发流

ScheduledJob 到点 → 调度器按 job_type 分派 handler：
  daily_report / weekly_report → 调 PydanticAI 报告函数 → 写 Report → 发 Notification(report_ready)
  reminder                     → 写 Notification(reminder)
  meeting_notice               → 写 Notification(meeting_notice)

Notification 产生 → 入应用内 inbox（Notification 表）
                 → 按 recipient 偏好外推飞书/钉钉 webhook（pushed_channels 防重复）

每个 handler 是独立的单一职责函数，互不调用、不协商——没有 Agent↔Agent。

F.3 通知通道

来源kind入口
定时任务触发reminder / meeting_notice / report_readyScheduledJob → Notification
webhook 即时事件mentioned / pr_reviewed直接写 Notification（不入 events_cache 主流程）
系统system集成连续失败 / 预算超 / 启动失败等

F.4 PM 看进展

PM 的"每日/每周看进展"不是新机制：cron ScheduledJob 触发 → DB 聚合查询当窗口全员活动 → 一次 PydanticAI 摘要 → 写团队 digest（Report，kind=daily/weekly，user_id=null 表团队级）。PM 也可在团队负载视图按需触发同一管道。

F.5 A → B 演进

⚠️ 进程内调度器在当前规模够用。真正撑不住的信号：后台 job 变多 + 高并发，定时任务拖累 web 响应（呼应附录 D 的 P95 > 1s 信号）。届时抽 capture/AI worker 出独立容器 + 上真正的任务队列（Arq / Celery / NATS），scheduled_jobs 表与触发流不变。现在别预建，留好 handler 函数边界即可。

附录 H：贡献投送与进度共享

让每人本地 AI(Claude Code / Codex / Cursor 等)的工作,经本人同意、按项目投送到平台,汇总成进度共享。来源:2026-05-28 brainstorm(详见 specs/2026-05-28-contribution-and-progress-design.md)。

H.1 隐私模型(全服务端 + 强隔离 + 同意门控)

  • 全服务端 个人助手/数据都在服务端;否决"本地优先沙箱"——权限隔离做扎实即可达隐私预期,不推翻现有架构。
  • 强隔离 chat / 通知仅本人可见,PM 永不可读(§5.4 已强制)。
  • 同意门控 本地 AI 工作不自动全量上传;投送 = 本人选择把哪条、归到哪个项目推上来,该动作即"同意";原始对话不离开私有区。

H.2 通用投送(source-agnostic,不绑定任何 agent)

# 认证:PAT(个人令牌,§5.5)
GET/POST/DELETE /me/tokens          # Authorization: Bearer pat_xxx

# 投送端点(任何能发 HTTP 的客户端都能用)
POST /me/contributions
  body: { summary, project_id?, kind? }   # kind: work | commit | note
  → 落 events_cache(source=agent, actor=本人, project_id=可空)

# events_cache 加 project_id(nullable, FK projects);EventSource 加 agent(VARCHAR,不用 PG 原生枚举)

H.3 跨 agent 客户端策略(押 MCP,不押 hook)

hook 不是跨 agent 标准——Claude Code(settings.json)、Codex(config.toml + notify)、Cursor 各不相同。跨 agent 一致的是 MCP。

客户端覆盖优先级
MCP server(contribute_work(project, summary))所有支持 MCP 的 agent(Claude Code/Cursor/Codex…),写一次通用主力
小 CLI teamplat contribute任何终端 agent 被叫去跑 / 人手动次
Git(已有 GitLab/GitHub 捕获)任何写代码的人,零 agent 要求已有底座
手动(助手 log_manual_work / 网页)没有任何本地 agent 的人已有
单 agent hook(如 Claude Code)仅特别在意某 agent 时可选

MCP server 作为独立组件(不在 FastAPI 主进程),带 PAT 调投送端点。

H.4 进度共享(share AI 简报)

share 页聚合:本项目任务(状态/负责人) + 投送的工作痕迹(events_cache 按 project_id) + 阻塞/评审信号 → AI 进展简报(已完成/进行中/阻塞风险/下一步),复用 decompose 的 PydanticAI 路子、记 llm_calls、按需生成(控成本)。v1 可先只基于任务数据,投送就绪后纳入 events。

H.5 范围

做:PAT、通用投送端点、events.project_id、MCP server、小 CLI、share AI 简报、归档折叠。不做(本轮):本地优先架构、per-agent hook 当主力、完整报告管线、enum→TEXT。

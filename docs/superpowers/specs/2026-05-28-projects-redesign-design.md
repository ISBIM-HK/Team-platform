## 0. 背景与目标

当前前端把"拆解 + 看板 + 助手"挤在一屏,且任务是**扁平挂在租户下**(无项目维度),所有任务混在一个看板里,看不清楚。

**目标**:引入真实"项目"维度,前端重构为**项目中心**的三栏工作区——每个项目有自己的看板、AI 方案、进度页;右侧助手常驻,随时可交互、也是创建项目的入口。

**非目标(本轮不做)**:enum→TEXT 重构、报告管线、项目级权限隔离(ACL)、对外公开分享链接。

## 1. 信息架构与布局

三栏:

```
┌─────────┬───────────────────────────────┬────────┐
│ 左:项目栏 │ 中:项目工作区                  │ 右:助手 │
│ · Tender │ [项目名]  看板 | AI方案 | 进度分享│ (固定) │
│ · 飞书   │ ───────────────────────────── │ 💬 对话 │
│ +新建项目 │  〔当前标签内容〕             │ 工具    │
│ ───      │                               │ [输入] │
│ 待处理建议│                               │        │
│ 成本(PM) │                               │ (窄屏收)│
└─────────┴───────────────────────────────┴────────┘
```

- **左 · 项目栏**:项目列表 + 全局入口(待处理建议、成本/PM 视图)+ "新建项目"
- **中 · 项目工作区**:选中项目后,三个标签页
  - **看板** — 本项目专属看板(5 列:待办/进行/阻塞/评审/完成),认领 + 状态机流转
  - **AI 方案** — 本项目的智能方案推荐(见 §4)
  - **进度分享** — 本项目只读进度/流程页(见 §5)
- **右 · 助手**:个人 AI 助手常驻,跨标签/页面始终可见;**窄屏(<1100px)可收起**为浮球
- 无选中项目时,中区显示项目总览/空态引导("跟右侧助手说一个目标,让它帮你建项目")

## 2. 数据模型

新增 `projects`,任务归属项目:

```
projects (
  id           UUID PK (v7)
  tenant_id    UUID NOT NULL → tenants
  name         TEXT NOT NULL
  description  TEXT
  status       TEXT  -- 'active' | 'archived'(应用层枚举校验,列用 VARCHAR)
  created_by   UUID → users
  created_at   TIMESTAMP
)

tasks 新增:
  project_id   UUID NOT NULL → projects   -- 每个任务必属一个项目
```

- **每租户一个"未分类(Inbox)"默认项目**:兜底手动/历史任务,保证 `project_id` 非空可落地
- decompose 生成的"目标父任务 + 子任务"全部落在**同一个项目**内(父任务仍用 `parent_task_id` 表达目标分组)
- `status` 等枚举本轮用 **VARCHAR + 应用层 Enum**(不引入新的 PG 原生枚举,避免日后加值的迁移痛;与既有 native enum 不一致是已知技术债,统一留给 enum→TEXT 那轮)

## 3. 项目创建流(走右侧助手)

沿用"AI 建议 → 人确认"基线:

```
你在右侧助手描述目标(如"做一个 Tender AI 合规校验功能")
   ↓
助手调用拆解 → 提议一个项目:{ 项目名, 描述, 子任务[] }
   ↓
中区/卡片展示该方案,可改名、增删/编辑子任务
   ↓
「确认创建」→ 落地:新建 Project + 其下父任务/子任务(project_id=新项目)
```

- **复用现有 `decompose` 建议类型**(不新增 PG 原生枚举值):accept 时,若 `target_ref` 带项目信息则**新建 Project + 其任务**;否则落到指定的已有项目
- 也保留 **"+ 新建空项目"**:手填名字直接建空项目,之后往里加任务或再拆解
- `target_ref` 扩展:携带 `project_name` / `project_id` 以区分"建新项目"还是"加入已有项目"

## 4. AI 方案推荐(项目页内)

项目工作区的"AI 方案"标签,聚合**本项目范围**的 AI 产出:
- 该项目的拆解方案(可重新生成 / 调整 / 采纳更多子任务)
- 本项目相关建议:智能分发(给子任务荐负责人)、重复任务检测、下一步建议
- 全部走 `ai_suggestions`(仅建议,人确认);低置信度(<0.6)不展示
- 每次 LLM 调用记 `llm_calls`(成本可观测,已有机制)

## 5. 进度分享页(每项目一个,只读)

面向团队同事"看进度 / 看流程"的只读视图:
- **进度**:状态分布(各列计数)、完成度 %、各成员负责的任务、最近更新
- **流程**:把该项目的目标→子任务拆解画成流程/时间线(父子 + 依赖)
- 路由 `/projects/{id}/share`(应用内);只读,不提供操作

## 6. 访问模型

- **团队(租户)内全可见**:同租户成员登录后可看所有项目及其进度页(延续现有"团队看板共享")
- **团队外的人**:不做对外公开链接;需要看就**给他建一个团队账号拉进来**(现有注册 / admin 建号)
- **不做项目级 ACL**(项目成员/隔离):留待未来真有跨团队隔离需求时再加

## 7. API 变更

| 端点 | 说明 |
|---|---|
| `GET /projects` | 列当前租户项目(含每项目任务计数/完成度概要) |
| `POST /projects` | 手建空项目 {name, description} |
| `GET /projects/{id}` | 项目详情 |
| `PATCH /projects/{id}` | 改名/描述/归档 |
| `GET /projects/{id}/tasks` | 该项目的任务(看板数据) |
| `GET /projects/{id}/share` | 进度/流程只读聚合(计数、完成度、成员、拆解流程) |
| `POST /decompose` | 调整:accept 在项目上下文落地(`target_ref` 含 `project_name` 则新建项目,含 `project_id` 则加入已有) |

- `GET /tasks` 仍保留(全局),但前端主要用 `/projects/{id}/tasks`
- 写操作鉴权延续 §5.4:member 改自己 owner/created 的任务,PM 全改

## 8. 后端改动

- `src/models/project.py`:Project 模型;`tasks` 加 `project_id`
- Alembic 迁移:建 `projects` 表 + `tasks.project_id`(先可空建列 → 回填 → 置非空,符合"无破坏迁移")
- `src/repositories/project_repo.py`:CRUD + 计数/概要查询
- `src/api/routes/projects.py`:上述端点
- 调整 `decompose`/`suggestions` accept:在项目上下文落地任务(复用 `decompose` 类型,`target_ref` 带 `project_name`/`project_id`;不新增 PG 枚举值)
- 每租户初始化"未分类"项目(注册建租户时一并建,或迁移时建)

## 9. 前端改动

- 重构 `frontend/` 为三栏:`项目栏 / 项目工作区(看板·AI方案·进度分享 三标签) / 固定助手`
- 沿用现有 Claude 暖色极简风(Fraunces + Hanken、陶土强调)
- 助手从"右侧固定列"保留,但窄屏收起为浮球
- 路由(前端内):`/`(项目总览)、`/p/{id}`(项目工作区,标签切换)、`/p/{id}/share`

## 10. 迁移

- 每租户建一个"未分类"项目
- 已有任务(含约 19 条测试数据)`project_id` 回填到该租户的"未分类"项目,不丢数据
- 迁移分步:加可空列 → 回填 → 改非空 + 外键

## 11. 范围边界

**做**:projects 实体 + 任务归属、三栏前端、项目工作区三标签、AI 方案(项目内)、进度分享页(团队内只读)、创建流(助手推荐+手建)、迁移。

**不做(本轮)**:enum→TEXT 全量重构、报告管线(归一化/日周报)、项目级 ACL、对外公开分享链接、时间戳 tz 重构。
